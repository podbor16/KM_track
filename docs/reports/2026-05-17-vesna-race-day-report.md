Пока шёл один цикл (10 минут), участники на сайте видели устаревшие данные.

**Причина:** `executemany(UPDATE)` в mysql-connector-python отправляет N отдельных SQL-запросов. При WAN-соединении до MySQL (79.174.89.159) каждый запрос занимал ~171мс. 895 участников × 171мс = 153с только для `_update_existing`. Плюс `_recalculate_ranks()` вызывался при каждом цикле (даже с 0 изменениями) и выполнял аналогичный объём запросов: ещё ~407с.

### 11:39 — Повторный --init → дубли в БД

При перезапуске был случайно запущен `--init` повторно. Таблица `results` не имеет UNIQUE KEY на `(event_id, start_number)`, поэтому `INSERT IGNORE` не защитил от дублирования.

**Последствия:** 1003 дублирующих записи. Устранено вручную:
```sql
DELETE r FROM results r
INNER JOIN (
  SELECT start_number, MAX(id) as max_id FROM results
  WHERE event_id = 106 GROUP BY start_number HAVING COUNT(*) > 1
) dups ON r.start_number = dups.start_number AND r.id = dups.max_id
WHERE r.event_id = 106
-- Affected rows: 1003
```

## Проведённые оптимизации

### 1. Локальный watchdog (`run_loader.py`)

Создан скрипт-обёртка с авто-перезапуском при падении. До этого loader запускался вручную.

**Результат:** loader перезапустился автоматически после крэша в 10:37 без участия организатора.

### 2. Оптимизация bulk-операций в БД

**Проблема:** `executemany(UPDATE)` = N отдельных SQL round-trips через WAN.

**Решение:** два новых метода в `RaceLoader`:
- `_bulk_upsert()` — один `INSERT...ON DUPLICATE KEY UPDATE` для всех строк (используется в `_update_existing`)
- `_bulk_update_join()` — один `UPDATE...INNER JOIN (SELECT...UNION ALL...)` для частичных обновлений (используется в `_recalculate_ranks`, `_recalculate_segment_ranks`)

**Результаты перф-теста на реальной БД:**

| Строк | executemany | bulk_upsert | bulk_update_join |
|-------|-------------|-------------|-----------------|
| 100 | 16.9s | 0.23s (**73x**) | 0.18s (**92x**) |
| 500 | 86.4s | 0.35s (**246x**) | 0.22s (**399x**) |
| 895 | 153.3s | 0.40s (**389x**) | 0.26s (**599x**) |

### 3. Условные пересчёты рангов

`_recalculate_ranks()` и `_recalculate_segment_ranks()` теперь вызываются только когда есть фактические изменения (`updated_r > 0` / `updated_s > 0`). Раньше — каждый цикл.

**Эффект:** циклы без изменений: ranks=0.00s вместо ~0.5–1s.

## Известные нерешённые проблемы

### 1. Нет UNIQUE KEY на (event_id, start_number) в таблице results

`INSERT IGNORE` в `_bulk_insert` не защищает от дублей. При повторном `--init` создаются дублирующие записи.

**Рекомендация:** добавить `ALTER TABLE results ADD UNIQUE KEY uq_event_dorsal (event_id, start_number)`.

### 2. Технические записи Copernico не в БД

Участники с номерами 1002, 1003, 1005, 1006, 1011, 1012, 1019, 1066 и "Зам-ий" присутствуют в Copernico но не в стартовом списке. Каждый цикл — предупреждения в логе.

**Рекомендация:** при `--init` вставлять все записи из Copernico, включая технические (с пометкой).

### 3. VPS loader не работает с Copernico API

Copernico API ограничивает доступ по IP. С VPS (89.108.88.104) — таймаут. Loader вынужден работать с ноутбука организатора.

**Рекомендация:** уточнить у Copernico возможность whitelisting IP сервера.
