## 2. Инструменты тестирования

| Инструмент | Версия | Назначение |
|------------|--------|------------|
| Locust | 2.43.4 | HTTP-нагрузка (GET/POST запросы) |
| asyncio SSE | Python 3.13 | SSE-нагрузка (persistent connections) |
| paramiko | — | Запуск SSE-теста на VPS через SSH |

**Методология:**
- HTTP-часть: Locust запускается с локальной машины, подключается через nginx/HTTPS
- SSE-часть: asyncio-скрипт загружается на VPS через SFTP и выполняется там (подключается к `localhost:8000`, минуя nginx)
- Оба теста работают **одновременно** в рамках одного прогона

## 5. Проблемы и решения в ходе тестирования

### 5.1 OOM при 335 curl-процессах
**Проблема:** 335 × curl (≈2 MB) + 4 × uvicorn workers (≈150 MB) = ~1.3 GB > 960 MB RAM → OOM killer убивал uvicorn.  
**Решение:** Заменили 335 curl-процессов одним asyncio-скриптом (~50 MB).

### 5.2 Слишком много DB-соединений
**Проблема:** `pool_size=20 × 4 workers = 80` соединений > MySQL `max_connections=64` → error 1040.  
**Решение:** `pool_size=5`, workers снижены с 4 до 2 → `5×2=10` соединений.

### 5.3 SSE timeout — 142/335 "no_connected"
**Проблема:** Первый прогон L1 — 142 VU не получили `:connected` за 30 с под комбинированной нагрузкой.  
**Решение:** Увеличили `deadline_connect` с 30 до 60 с в asyncio SSE-скрипте.

### 5.4 Медленный HTTP при совместной нагрузке (avg 6 300 мс)
**Проблема:** FastAPI с `response_model=RaceResultsResponse` выполняет Pydantic-валидацию и `jsonable_encoder` синхронно в event loop на **каждый** запрос (~50 мс/запрос). При 50 concurrent → последний ждёт ~2.5 с.  
**Решение:** Введён `_json_cache` — JSON-строка сериализуется **один раз** при построении кеша, HTTP-endpoint возвращает `Response(content=cached_json)` без re-validation.  
**Эффект:** avg время 50 concurrent запросов: **2 700 мс → 280 мс** (×10).

## 6. Анализ пределов нагрузки

### Почему L1 прошёл, а L2 нет

| Ресурс | Норма (idle) | При L1 | При L2 |
|--------|-------------|--------|--------|
| RAM | ~846 MB | ~870 MB | OOM risk |
| CPU | ~10% | ~80–95% | >100% (очередь) |
| DB соединений | 0–3 | 5–8 | 10+ (пул исчерпан) |
| Event loop latency | <10 мс | ~50 мс | >500 мс |

При L2 на 1 CPU: asyncio event loop 2 workers + asyncio SSE-клиент (1335 корутин) конкурируют за единственный CPU. Планировщик не успевает обслуживать все корутины, SSE `:connected` не отправляется в течение 60 с → VU помечает "no_connected".

### Требования для L2 (рекомендуемые)

| Компонент | Текущий | Нужен для L2 |
|-----------|---------|--------------|
| CPU | 1 ядро | 4 ядра |
| RAM | 960 MB | 4 GB |
| uvicorn workers | 2 | 4–8 |
| DB max_connections | 64 | 128 |
| Swap | нет | 2 GB |

Все оптимизации сохранены в production-коде:

1. **`app.py`** — `pool_size=5` (было 20)
2. **`deploy/km_track.service`** — `--workers 2` (было 4), `LimitNOFILE=65535`
3. **`src/tracker/services/results_service.py`** — `_json_cache: dict` — кеш сериализованного JSON
4. **`src/tracker/routers/api.py`** — `/api/event-results` без `response_model`, fast path через `_json_cache`
5. **`tests/load/sse_load_remote.py`** — `deadline_connect = start + 60` (было 30)
6. **`/etc/nginx/nginx.conf`** — `worker_connections 65535`

## 8. Вывод

**KM_track успешно прошёл L1 (500 одновременных пользователей)** на VPS-стенде c 1 CPU / 960 MB RAM:
- 0 HTTP-ошибок за 8 минут при 13 613 запросах
- 335/335 SSE-соединений (100%) удержаны 8+ минут
- Среднее время ответа 1.4 с (приемлемо при внешней БД)

Для поддержки L2 (2 000 пользователей) и выше необходима горизонтальная масштабируемость: несколько VPS за балансировщиком, shared Redis для кеша, либо более мощный dedicated сервер (4+ CPU, 4+ GB RAM).
