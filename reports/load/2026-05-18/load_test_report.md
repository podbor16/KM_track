## 5. Проблемы и решения в ходе тестирования

### 5.1 OOM при 335 curl-процессах
**Проблема:** 335 × curl (≈2 MB) + uvicorn workers → OOM killer убивал uvicorn.  
**Решение:** Заменили 335 curl-процессов одним asyncio-скриптом (~50 MB).

### 5.2 Слишком много DB-соединений
**Проблема:** `pool_size=20 × 4 workers = 80` соединений > MySQL `max_connections=64`.  
**Решение:** `pool_size=5`, workers снижены с 4 до 2 → `5×2=10` соединений.

### 5.3 SSE timeout — 142/335 "no_connected"
**Проблема:** 142 VU не получили `:connected` за 30 с при совместной нагрузке.  
**Решение:** `deadline_connect` увеличен с 30 до 60 с в asyncio SSE-скрипте.

### 5.4 Медленный HTTP при совместной нагрузке (avg 6 300 мс)
**Проблема:** Pydantic-валидация + `jsonable_encoder` на каждый запрос (~50 мс), 50 concurrent → очередь.  
**Решение:** `_json_cache` — JSON-строка сериализуется один раз, endpoint возвращает `Response(content=cached_json)`.  
**Эффект:** avg 50 concurrent: **2 700 мс → 280 мс** (×10).

### 5.5 paramiko socket timeout при `apt-get install mysql-server`
**Проблема:** `exec_command(timeout=300)` прерывал установку раньше завершения.  
**Решение:** `channel.settimeout(None)` + deadline-based polling loop.

## 6. Анализ пределов нагрузки

### Почему L2 SSE падает на 2 CPU / 2 GB

1335 SSE-соединений = 1335 независимых корутин, каждая периодически делает запрос в БД. При pool_size=10 суммарно это создаёт очередь к DB-пулу, которую event loop не успевает обслуживать — 317 клиентов не получают `:connected` в течение 60 с.

| Ресурс | При L1 (500 users) | При L2 (2000 users) |
|--------|-------------------|---------------------|
| SSE connected | 335/335 (100%) | 1018/1335 (76%) |
| HTTP avg | 1 394 мс | 3 435 мс |
| DB connections | 5–8 / 10 | 10/10 (пул исчерпан) |
| SSE одновременно | 335 | ~1000 (практический предел) |

**Практический предел** текущей конфигурации: **~1000 SSE + 665 HTTP** при приемлемом времени ответа.

### Пути до L2 PASS

| Вариант | Стоимость | Сложность | Эффект |
|---------|-----------|-----------|--------|
| **Redis Pub/Sub для SSE** | ~0 ₽ (Redis на VPS) | Средняя | Убирает N×DB → 1×DB, позволяет 2000+ SSE на текущем железе |
| Апгрейд до 4 CPU / 4 GB | ~1 500 ₽/мес | Минимальная | Прямолинейное масштабирование |
| Горизонтальное (2 VPS + балансировщик) | ~1 000 ₽/мес × 2 | Высокая | Требует Redis + sticky sessions |

**Рекомендация:** Redis Pub/Sub — оптимальный следующий шаг. Снижает нагрузку на DB O(N)→O(1), позволяет пройти L2 без дополнительных расходов и является фундаментом для горизонтального масштабирования.

## 8. Вывод

**KM_track уверенно проходит L1 (500 пользователей)** на любой конфигурации: 0 HTTP-ошибок, 335/335 SSE (100%), avg 1.4 с.

**L2 (2000 пользователей) на 2 CPU / 2 GB:**
- HTTP: фактически пройден — 38 945 запросов, 0.02% ошибок, 87 RPS
- SSE: 1018/1335 (76%) — не достигает порога 95%
- Практический предел: ~1000 SSE-соединений одновременно

**Следующий шаг для L2 PASS:** Redis Pub/Sub — один publisher читает БД, все SSE-клиенты получают события через Redis channel. Это устраняет бутылочное горлышко DB-пула без изменения железа.
